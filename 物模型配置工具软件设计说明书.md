# 物模型配置工具软件设计说明书

## 1. 文档说明

### 1.1 编写目的

本文档用于说明 EMS 物模型配置工具的软件设计方案，作为产品设计、前端实现、测试验证、后续迭代和交付说明的依据。

### 1.2 软件定位

本工具是一个基于浏览器运行的单页式物模型配置向导，面向 EMS 场景下的设备或站点建模工作，帮助用户按步骤完成以下配置：

1. 物模型介绍浏览
2. 档案信息配置
3. 指令配置
4. 事件配置
5. 属性配置
6. 行为自动生成与启停控制
7. 预览与导出

### 1.3 适用范围

本文档适用于以下对象：

1. 产品经理和需求分析人员
2. 前端开发人员
3. 测试人员
4. 项目实施与交付人员
5. 后续扩展该工具的维护人员

### 1.4 参考资料

1. 当前实现文件：thing-model-wizard.html
2. 领域说明资料：物模型说明.md
3. 本轮需求迭代中的交互、字段、规则和导出约束

## 2. 软件概述

### 2.1 建设背景

在 EMS 与能源物联网场景中，不同设备、站点和通信协议需要通过统一的物模型进行抽象描述。物模型需要覆盖设备或站点的静态档案信息、数据属性、可执行行为、事件规则以及南向通信指令。为降低建模门槛，提高配置效率，设计本工具作为可视化配置入口。

### 2.2 建设目标

本工具的设计目标如下：

1. 提供清晰的向导式配置流程，降低建模复杂度。
2. 支持设备与集群两类档案对象建模。
3. 支持多接口类型、多协议类型的档案建模入口。
4. 在当前版本中，以 Modbus 为核心实现结构化指令配置规则。
5. 支持属性、行为、事件之间的关联配置。
6. 支持属性 CSV 导入导出，提高批量建模效率。
7. 支持物模型 JSON / YAML 文件导入。
8. 支持最终导出 JSON 和 YAML 两种格式的物模型文件。

### 2.3 当前版本范围

当前版本是单文件前端原型工具，特点如下：

1. 采用纯前端实现，无后端服务依赖。
2. 数据仅在页面内存中维护，用户通过导出文件完成结果持久化。
3. 多协议可选，但结构化指令新增当前仅支持 Modbus TCP 和 Modbus RTU。
4. 档案信息页面支持整模型 JSON / YAML 导入。
5. 属性寄存器规则当前按 Modbus 场景设计。

## 3. 总体设计

### 3.1 设计原则

1. 向导化：将复杂建模任务拆分为多个步骤逐步完成。
2. 列表化：指令、属性、事件采用表格列表进行统一管理。
3. 规则化：将 Modbus 指令、属性自动生成、行为自动生成等逻辑固化为规则。
4. 可导出：所有页面内配置最终转换为标准结构化输出。
5. 可扩展：通过常量配置和函数分段实现协议、字段、规则扩展。

### 3.2 技术架构

本工具采用单页前端架构，主要技术组成如下：

1. HTML5：页面结构和表单定义。
2. CSS：结合 Bootstrap 实现基础布局、响应式和样式。
3. Bootstrap 5.3.2：提供表单、表格、模态框、按钮和基础响应式栅格。
4. Bootstrap Icons 1.11.3：提供步骤图标和操作图标。
5. 原生 JavaScript：实现页面状态管理、规则计算、校验、导入导出和事件绑定。
6. js-yaml 4.1.0：实现 YAML 预览与导出。

### 3.3 运行形态

工具以单个 HTML 文件运行。用户打开页面后，全部交互均在浏览器本地完成，不依赖服务端接口。

### 3.4 状态管理模型

系统核心状态对象为 `thingModel`，用于承载全部建模结果。当前初始化时会自动带出默认 `online` 属性和默认通信状态事件 `event_0`。

```js
const thingModel = {
  profile: { name: '', description: '', modelType: '', thingType: '', interfaceType: '', protocolType: '' },
  properties: [
    {
      id: 'online',
      name: '在线状态',
      description: '0,离线;1,在线',
      dataType: 'bool',
      accessMode: 'r',
      instructionId: '',
      eventId: 'event_0',
      registerBaseAddress: null,
      registerAddress: null,
      registerOffset: null,
      bitOffset: null,
      autoGenerated: false
    }
  ],
  sessions: [],
  actions: [],
  events: [
    {
      id: 'event_0',
      name: '设备通讯离线',
      description: '设备通讯状态',
      level: 'fault',
      type: 'eq',
      triggerThreshold: '0',
      deadband: null
    }
  ]
};
```

同时，通过 `currentStep` 管理当前向导步骤，通过 `TOTAL_STEPS = 7` 管理流程总步数。

## 4. 功能结构设计

### 4.1 七步流程

当前工具共设计 7 个步骤：

| 步骤 | 名称 | 作用 |
| --- | --- | --- |
| 1 | 物模型介绍 | 说明物模型、档案、指令、属性、行为、事件的概念 |
| 2 | 档案信息 | 配置对象身份、接口和协议，并支持导入整份物模型 |
| 3 | 指令 | 配置设备交互指令，当前重点支持 Modbus |
| 4 | 事件 | 配置事件规则 |
| 5 | 属性 | 配置属性元数据、寄存器配置、关联信息 |
| 6 | 行为 | 根据属性读写标志自动生成行为 |
| 7 | 预览导出 | 查看最终输出并导出 JSON/YAML |

### 4.2 页面导航设计

页面导航采用以下设计：

1. 顶部步骤条展示整体流程。
2. 左右圆形箭头按钮用于切换上一步、下一步。
3. 底部步骤计数器显示“第 N 步 / 共 7 步”。
4. 到达最后一步前一页时，下一步按钮切换为完成态。

### 4.3 模态框设计

以下模块通过模态框完成新增和编辑操作：

1. 属性配置模态框
2. 事件配置模态框
3. 指令配置模态框

采用模态框的原因是：

1. 保持主流程列表页面简洁。
2. 有利于单条记录的表单校验。
3. 方便在新增与编辑之间复用同一套 UI 与逻辑。

## 5. 功能模块设计

### 5.1 物模型介绍模块

#### 5.1.1 功能目的

用于向用户说明物模型配置的业务背景和配置流程，降低首次使用的理解成本。

#### 5.1.2 设计内容

本模块说明以下概念：

1. 档案信息 Profile
2. 指令 Command
3. 属性 Property
4. 行为 Action
5. 事件 Event
6. 配置流程

#### 5.1.3 设计特点

1. 使用卡片式布局说明各模块含义。
2. 强调该工具面向 EMS 与能源物联网场景。
3. 通过步骤概览降低用户对后续流程的陌生感。

### 5.2 档案信息模块

#### 5.2.1 功能目的

用于描述设备或站点的身份信息和通信入口信息。

#### 5.2.2 配置字段

1. 名称
2. 描述
3. 档案类型
4. 物类型
5. 接口类型
6. 协议类型

#### 5.2.3 关键规则

1. 名称为必填项。
2. 档案类型支持 `device` 和 `cluster`。
3. 物类型受档案类型控制：未选择档案类型时不可选。
4. 接口类型支持 `Serial`、`CAN`、`Network`。
5. 协议类型受接口类型控制。

#### 5.2.4 协议选项设计

当前协议选项包括但不限于：

1. 串口：Modbus RTU、iec103、DL/T 645-1997、DL/T 645-2007、自定义协议
2. 网络：Modbus TCP、Modbus RTU、iec103、iec104、iec61850、OPC UA、MQTT、HTTP/HTTPS、自定义协议
3. CAN：CAN 协议、自定义协议

#### 5.2.5 物模型导入设计

档案信息页面提供“导入物模型”按钮，支持以下能力：

1. 支持导入 JSON 和 YAML 两种格式。
2. 导入内容覆盖当前内存中的 `profile`、`sessions`、`events`、`properties`、`actions`。
3. 当当前页面已经存在配置时，导入前会弹出覆盖确认。
4. 导入完成后，会同步刷新档案表单、指令列表、事件列表、属性列表、行为列表和预览内容。

### 5.3 指令模块

#### 5.3.1 功能目的

用于管理物模型与南向设备或系统之间的数据交互指令。

#### 5.3.2 设计方式

1. 指令采用列表化管理。
2. 支持手动新增、编辑、删除。
3. 当前结构化规则仅实现 Modbus 指令。

#### 5.3.3 指令字段

手动新增指令时，支持以下字段：

1. 指令名称
2. 指令描述
3. 设备地址
4. 功能码
5. 寄存器起始地址
6. 寄存器数量

#### 5.3.4 Modbus 规则

当前支持的功能码包括：

1. `0x1` 读线圈
2. `0x2` 读离散输入
3. `0x3` 读保持寄存器
4. `0x4` 读输入寄存器

#### 5.3.5 指令校验规则

1. 设备地址非必填，默认值为 `1`。
2. 设备地址允许范围为 `1` 到 `247`。
3. 起始地址支持十进制与 `0x/0X` 开头十六进制输入。
4. 寄存器数量按功能码分别校验：
   1. `0x1` / `0x2` 范围为 `1-2000`
   2. `0x3` / `0x4` 范围为 `1-125`

#### 5.3.6 当前限制

1. 非 Modbus 协议暂未实现结构化指令录入。
2. 对于其他协议，目前仅提供档案级选择与说明文本，不提供同等深度的指令规则生成。

### 5.4 属性模块

#### 5.4.1 功能目的

用于对物模型中的属性进行列表化管理，并建立属性与指令、事件及寄存器配置之间的关系。

#### 5.4.2 列表化管理能力

属性模块支持以下能力：

1. 手动新增属性
2. 编辑属性
3. 删除属性
4. 自动生成属性
5. CSV 导入
6. CSV 导出

#### 5.4.3 属性配置分类

属性配置分为两大类：

1. 数据相关属性
2. 寄存器配置

#### 5.4.4 数据相关属性字段

当前支持以下字段：

1. 属性 ID
2. 名称
3. 描述
4. 数据类型
5. 单位
6. 基数
7. 系数
8. 最大值
9. 最小值
10. 精度
11. 读写属性
12. 数据质量
13. 指令关联
14. 事件关联

说明：数据质量字段已在界面中隐藏，但在属性对象和最终导出的物模型文件中仍然保留。

#### 5.4.5 数据类型设计

当前支持以下属性数据类型：

1. 有符号整数 `signedInt`
2. 无符号整数 `unsignedInt`
3. 有符号浮点数 `signedFloat`
4. 无符号浮点数 `unsignedFloat`
5. 位图 `bitmap`
6. 枚举 `enum`
7. 布尔 `bool`
8. 字符串 `string`

#### 5.4.6 精度规则

1. 精度仅对浮点类型启用。
2. 当前支持 `1`、`2`、`3` 位小数。
3. 默认精度为 `2` 位小数。
4. 非浮点类型下，精度字段自动禁用且不参与保存。

#### 5.4.7 关联设计

属性支持两类关联：

1. 指令关联：用于绑定某个指令 ID，便于管理属性与南向读写指令之间的关系。
2. 事件关联：用于绑定某个事件 ID，例如 `event_1`，便于管理属性与事件规则之间的关系。

#### 5.4.8 寄存器配置字段

当前属性寄存器配置按 Modbus 设计，支持以下字段：

1. 寄存器基地址
2. 寄存器地址
3. 数据大小
4. 字节顺序
5. 寄存器偏移
6. 位偏移

#### 5.4.9 寄存器地址计算规则

当前 Modbus 映射规则如下：

1. 当属性绑定指令后，寄存器基地址跟随指令起始地址自动带出。
2. 寄存器地址由用户填写。
3. 寄存器偏移由系统自动计算。
4. 计算公式为：`寄存器偏移 = 寄存器地址 - 寄存器基地址`。
5. 若绑定了合法指令，则寄存器地址必须落在该指令请求范围内。
6. 未绑定指令时，协议相关字段可保持空值，不参与 Modbus 校验。

工具会在属性编辑界面中自动显示上述结果，并在列表、导出和属性自动生成中统一使用有效寄存器地址。

#### 5.4.10 属性自动生成规则

属性自动生成规则基于指令列表：

1. 扫描结构化 Modbus 指令。
2. 按每条指令的寄存器数量生成默认属性数。
3. 属性 ID 自动命名为 `auto_n`。
4. 属性名称自动命名为“属性点 n”。
5. 默认数据类型为 `signedInt`。
6. 默认读写属性由功能码推导：
   1. 读类功能码生成只读属性
   2. 写类功能码生成只写属性
7. 自动写入指令关联。
8. 自动写入寄存器基地址、寄存器地址、寄存器偏移、默认数据大小和默认字节顺序。
9. 自动生成后的属性仍允许用户继续编辑数据相关字段和协议相关字段。
10. 新建配置默认自带 `online` 属性，该属性不属于属性批量自动生成结果。

#### 5.4.11 CSV 导入导出设计

属性模块支持 CSV 导入导出，用于批量建模。

CSV 当前覆盖字段包括：

1. 基础元数据字段
2. 指令关联与事件关联字段
3. 寄存器基地址、寄存器地址、寄存器偏移、位偏移等寄存器字段
4. 自动生成标记

CSV 导入规则包括：

1. 表头必须包含必要字段。
2. 属性 ID 必须合法且不可重复。
3. 数据类型、读写属性、字节顺序、数据大小必须在允许范围内。
4. 数值字段必须满足数值格式和范围约束。
5. 已绑定指令的属性会校验寄存器地址是否落在指令请求范围内。
6. 未绑定指令的属性不得填写协议相关字段。
7. 兼容旧 CSV 中仅提供寄存器地址而未提供寄存器基地址的情况，系统会尽量推导基地址。

### 5.5 行为模块

#### 5.5.1 功能目的

用于根据属性的读写能力自动生成对外行为集合。

#### 5.5.2 生成规则

1. 页面提供“自动生成”按钮，用于根据当前属性列表刷新行为集合。
2. 对 `r` 或 `rw` 属性生成 `GET` 行为。
3. 对 `w` 或 `rw` 属性生成 `SET` 行为。
4. `rw` 属性会同时生成 `GET` 与 `SET` 两类行为。
5. 行为 ID 分别采用 `getXxx` 和 `setXxx` 形式。
6. 用户可对自动生成行为进行启用或禁用。
7. 若整模型导入文件中包含行为列表，则会按行为 ID 恢复启用状态。
8. 最终导出时，仅输出启用状态的行为。

### 5.6 事件模块

#### 5.6.1 功能目的

用于描述设备运行过程中的告警、故障或提示规则。

#### 5.6.2 配置字段

1. 事件 ID
2. 事件名称
3. 事件描述
4. 事件等级
5. 事件类型
6. 触发阈值
7. 死区

#### 5.6.3 关键规则

1. 事件 ID 自动生成，格式为 `event_x`。
2. 事件等级支持：提示、告警、故障。
3. 事件类型支持：等值、越上限、越下限。
4. 越上限和越下限事件支持死区配置，默认值为 `2`。
5. 事件与属性的关联关系由属性侧的“事件关联”字段维护。
6. 新建配置默认自带 `event_0`，用于表达设备通讯状态事件。

### 5.7 预览与导出模块

#### 5.7.1 功能目的

用于展示当前物模型结果并导出文件。

#### 5.7.2 功能内容

1. 统计卡片摘要展示
2. JSON 预览
3. YAML 预览
4. JSON 下载
5. YAML 下载
6. 一键重新配置

#### 5.7.3 导出规则

导出内容包括：

1. `profile`
2. `properties`
3. `sessions`
4. `actions` 中已启用的行为
5. `events`

## 6. 数据结构设计

### 6.1 Profile 结构

```json
{
  "name": "",
  "description": "",
  "modelType": "device|cluster",
  "thingType": "",

  "protocolType": ""
```

```json
{
  "id": "cmd_1",
  "name": "",
  "description": "",
  "request": "",
  "response": "",
  "deviceAddress": 1,
  "functionCode": "03",
  "startAddress": 0,
  "registerCount": 1,
  "protocol": "Modbus TCP|Modbus RTU",
  "commandType": "modbus"
}
```

### 6.3 Property 结构

```json
{
  "id": "online",
  "name": "在线状态",
  "description": "0,离线;1,在线",
  "dataType": "bool",
  "unit": "",
  "base": 0,
  "coefficient": 1,
  "minValue": "",
  "maxValue": "",
  "precision": null,
  "accessMode": "r|w|rw",
  "quality": "",
  "instructionId": "",

  "registerBaseAddress": null,
  "registerAddress": null,
  "dataSize": "",
  "byteOrder": "",
  "registerOffset": null,
```

说明：

1. 协议相关字段在未绑定指令时可为空。
2. 新建配置默认创建上述 `online` 属性。
3. 默认 `online` 属性不配置协议相关字段。

## 7. 核心规则与校验设计

### 7.1 标识符规则

1. 属性 ID 和事件 ID 必须以字母或下划线开头。
2. 属性 ID 和事件 ID 仅允许字母、数字、下划线。
3. 同类型对象内 ID 不能重复。

### 7.2 联动规则

1. 档案类型决定物类型选项。
2. 接口类型决定协议类型选项。
3. 协议类型会影响属性寄存器字段是否启用。
4. 属性模块的指令关联依赖指令列表。
5. 属性模块的事件关联依赖事件列表。
6. 行为模块依赖属性列表自动生成。
7. 新建配置默认带出 `online` 属性和 `event_0` 事件。

### 7.3 类型规则

1. 浮点类型可设置精度。
2. 非浮点类型禁用精度设置。
3. 数据大小、字节顺序、位偏移适用于寄存器映射场景。

### 7.4 地址规则

1. 起始地址、寄存器基地址支持十进制和十六进制。
2. 寄存器地址计算结果不得超过 `0xFFFF`。
3. 位偏移范围为 `0-15`。
4. 寄存器偏移必须大于等于 `0`。

### 7.5 输出过滤规则

1. 行为列表在导出时仅输出 `enabled = true` 的项。
2. 属性列表会完整导出，包括界面隐藏的 `quality` 字段。
3. 预览结果与导出文件结构保持一致。

## 8. 界面与交互设计

### 8.1 界面布局

界面采用以下布局方式：

1. 顶部标题区
2. 顶部步骤条
3. 中央向导卡片
4. 两侧步骤切换箭头
5. 下方步骤计数器

### 8.2 展示方式

1. 向导主体使用卡片和表格展示。
2. 新增与编辑使用模态框。
3. 空列表使用空状态提示。
4. 预览区域使用代码风格展示导出内容。

### 8.3 反馈方式

当前交互反馈方式包括：

1. 表单必填校验提示
2. 非法值即时错误提示
3. 自动生成前确认提示
4. 删除前确认提示
5. 导入覆盖确认与导入结果提示

### 8.4 响应式设计

通过 Bootstrap 栅格和媒体查询支持桌面与移动端浏览。移动端下会简化步骤标签显示，并压缩边距和导航按钮尺寸。

## 9. 非功能设计

### 9.1 易用性

1. 向导化流程降低理解成本。
2. 列表化管理便于批量维护。
3. CSV 导入导出适合工程场景下的大批量建模。

### 9.2 可维护性

1. 采用常量集中管理枚举值和标签。
2. 采用分段函数组织不同步骤逻辑。
3. 状态对象集中管理，便于后续迁移到更复杂框架。

### 9.3 可扩展性

当前工具具备以下扩展基础：

1. 可扩展接口和协议枚举。
2. 可扩展非 Modbus 协议的指令规则。
3. 可扩展更细粒度的数据类型配置。
4. 可扩展更复杂的属性-事件-行为联动校验。

### 9.4 性能特性

由于当前为纯前端本地页面，且数据量通常为单个设备或站点级别，整体性能开销较小，主要消耗来自：

1. 列表渲染
2. CSV 解析
3. 预览序列化

在当前规模下，可满足原型和配置工具使用需求。

## 10. 已知限制

### 10.1 协议实现限制

1. 多协议可选，但结构化指令录入当前仅支持 Modbus TCP / Modbus RTU。
2. 属性寄存器配置当前按 Modbus 规则建模。

### 10.2 一致性限制

1. 属性侧的事件关联是当前唯一关联入口，事件对象本身不保存 `propertyId`。
2. 指令侧与属性侧关联存在业务关联，但当前并未实现全量交叉一致性检查。

### 10.3 数据表达限制

1. 当前 `enum`、`bitmap`、`bool`、`string` 仅完成类型级支持，未提供更细的取值域编辑器。
2. 数据质量字段仅保留在数据对象中，界面上不展示。
3. 尚未实现本地草稿持久化和版本管理。

## 11. 后续建议

### 11.1 功能建议

1. 为非 Modbus 协议补充结构化指令编辑规则。
2. 为 `enum` 和 `bitmap` 补充值域与位定义编辑界面。
3. 增加组件 Component 建模能力。
4. 增加导入模板下载与字段说明。
5. 增加属性、事件、行为、指令之间的一致性校验报告。

### 11.2 工程建议

1. 如后续功能继续增长，可将单文件页面拆分为模块化前端工程。
2. 可引入持久化存储或后端服务，实现项目保存与再次编辑。
3. 可为导出物模型增加版本号、元信息和校验摘要。

## 12. 结论

本工具当前已经形成一套面向 EMS 场景的物模型配置前端方案，具备向导化配置、规则化生成、列表化管理、CSV 批量处理、整模型 JSON/YAML 导入以及 JSON/YAML 导出的核心能力。就当前版本而言，重点实现了 Modbus 场景下的指令、事件、属性、行为建模链路，并为后续协议扩展、组件扩展和工程化演进预留了清晰的设计基础。
